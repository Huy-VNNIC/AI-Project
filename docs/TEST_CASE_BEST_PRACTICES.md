## Introduction

Effective test cases ensure software quality, prevent regressions, and document expected behavior. This guide covers best practices for designing, writing, and maintaining test cases—especially for test cases generated by our AI system.

### Why Test Cases Matter

- **Quality Assurance** - Catch bugs before production
- **Documentation** - Describe system behavior
- **Regression Prevention** - Ensure updates don't break existing features
- **Automation** - Enable continuous testing
- **Confidence** - Verify requirements are met

## Test Case Design Principles

### 1. **Clear Objective**

Every test case should have ONE clear purpose.

**✅ GOOD:**
```
Title: Login with correct credentials succeeds
Purpose: Verify system accepts valid email/password combination
```

**❌ POOR:**
```
Title: Login test
Purpose: Test login and profile update
```

### 2. **Independence**

Test cases should NOT depend on each other.

**✅ GOOD:**
```
Test 1: Create user account
Test 2: Log in with created account (setup new user in Test 2)
Test 3: Update user password (setup new user in Test 3)
```

**❌ POOR:**
```
Test 1: Create user account
Test 2: Log in with user from Test 1
Test 3: Update password from Test 2
(If Test 1 fails, tests 2 & 3 fail too)
```

### 3. **Repeatability**

Same test with same input should produce same result.

**✅ GOOD:**
```
Test: Upload file with specific timestamp
Expected: File stored with correct timestamp
(Can run multiple times, always same result)
```

**❌ POOR:**
```
Test: Upload current file
Expected: File is processed
(Result depends on current state—not repeatable)
```

### 4. **Clarity**

Anyone should understand what the test does.

**✅ GOOD:**
```
Step 1: Navigate to login page
Step 2: Enter email "user@example.com"
Step 3: Enter password "Password123!"
Expected: User dashboard displayed
```

**❌ POOR:**
```
Step 1: Go to page
Step 2: Enter stuff
Step 3: Check results
```

### 5. **Isolation**

Test should NOT affect other tests.

**✅ GOOD:**
```
Test: Create user in test database
Cleanup: Delete created user after test
(Other tests unaffected)
```

**❌ POOR:**
```
Test: Create user in production database
(Affects other tests, leaves data behind)
```

## Test Case Structure

### Standard Format

```
Test Case ID: TC-001
Title: [Short description of test]
Category: [Authentication, Payment, etc.]
Priority: [High, Medium, Low]

Preconditions:
- User account exists
- User not logged in

Steps:
1. Navigate to login page
2. Enter email: user@example.com
3. Enter password: Password123!
4. Click "Sign In" button

Expected Results:
- User dashboard displays
- Welcome message shows user name
- Session cookie created

Postconditions:
- User logged in and active
```

### Component Breakdown

| Component | Purpose | Example |
|-----------|---------|---------|
| **Test Case ID** | Unique identifier | TC-LOGIN-001 |
| **Title** | Brief description | Login with valid credentials |
| **Category** | Functional area | Authentication |
| **Priority** | Test importance | High (critical), Medium, Low |
| **Preconditions** | Setup needed | User account must exist |
| **Steps** | Actions to perform | Click button, enter text, etc. |
| **Expected Results** | What should happen | Page loads, message displays |
| **Postconditions** | Final state | User logged in, session created |

## Comprehensive Test Coverage

### Test Categories

#### 1. **Happy Path Tests**
Tests normal, expected user flow.

```
Test: User logs in with valid credentials
Step 1: Enter correct email
Step 2: Enter correct password
Expected: Login successful, dashboard displayed
```

#### 2. **Negative Tests**
Tests error conditions and invalid inputs.

```
Test: User logs in with invalid password
Step 1: Enter valid email
Step 2: Enter incorrect password
Expected: Error message displayed, login rejected
```

#### 3. **Boundary Tests**
Tests edge cases and limits.

```
Test: Upload 10MB file (exactly at size limit)
Expected: Upload succeeds

Test: Upload 10.1MB file (over size limit)
Expected: Upload rejected with error
```

#### 4. **Security Tests**
Tests security constraints.

```
Test: SQL injection attack on login
Step 1: Enter "admin' OR '1'='1"
Expected: Treated as normal input, rejected
```

#### 5. **Integration Tests**
Tests interactions between components.

```
Test: Payment processing integration
Step 1: User selects items
Step 2: Proceeds to payment
Step 3: System contacts payment gateway
Expected: Order created, payment confirmed
```

#### 6. **Performance Tests**
Tests response times and scalability.

```
Test: Search with 100,000 records
Expected: Results returned within 2 seconds
```

### Coverage Matrix Example

| Feature | Happy Path | Negative | Boundary | Security | Integration |
|---------|-----------|----------|----------|----------|-------------|
| Login | TC-001 | TC-002, 003 | - | TC-004 | TC-005 |
| Upload | TC-010 | TC-011, 012 | TC-013 | TC-014 | TC-015 |
| Payment | TC-020 | TC-021 | TC-022 | TC-023 | TC-024 |
| Search | TC-030 | TC-031 | TC-032 | - | TC-033 |

## Common Patterns

### Pattern 1: Create-Read-Update-Delete (CRUD)

```
CREATE TEST:
Test: Create new product
Step 1: Navigate to product creation form
Step 2: Enter product name "Widget"
Step 3: Enter price "19.99"
Step 4: Click Save
Expected: Product created, ID assigned, confirmation shown

READ TEST:
Test: Find created product
Step 1: Search for "Widget"
Expected: Product found, details displayed correctly

UPDATE TEST:
Test: Update product price
Step 1: Open product "Widget"
Step 2: Change price to "24.99"
Step 3: Click Save
Expected: Price updated, confirmation shown

DELETE TEST:
Test: Delete product
Step 1: Open product "Widget"
Step 2: Click Delete
Step 3: Confirm deletion
Expected: Product removed, no longer searchable
```

### Pattern 2: Form Validation

```
Test: Email field validation
- Empty: Should show "Required" error ✓
- Invalid format (abc): Should show "Invalid email" ✓
- Valid format (user@domain.com): Should accept ✓
- Too long (100+ chars): Should show "Too long" ✓
- Special chars only (@#$): Should reject ✓
```

### Pattern 3: Login/Logout Flow

```
PRE-LOGIN:
- Test invalid credentials rejected
- Test account locked after 5 attempts

POST-LOGIN:
- Test user can access protected resources
- Test session timeout after inactivity

POST-LOGOUT:
- Test user redirected to login page
- Test protected resources unavailable
```

### Pattern 4: Pagination

```
Test: Page navigation with 100 records
- First page: Items 1-10, "Previous" disabled ✓
- Second page: Items 11-20, both buttons enabled ✓
- Last page: Items 91-100, "Next" disabled ✓
- Jump to page 5: Items 41-50 ✓
```

## Test Case Examples

### Example 1: Authentication

```
Test Case ID: TC-AUTH-001
Title: User login with valid credentials succeeds
Priority: High

Preconditions:
- User account "user@example.com" exists with password "SecurePass123!"
- User not currently logged in
- Login page is accessible

Steps:
1. Navigate to https://app.example.com/login
2. Enter "user@example.com" in email field
3. Enter "SecurePass123!" in password field
4. Click "Sign In" button
5. Wait for page redirect

Expected Results:
- Login form closes
- User dashboard displays
- Welcome message shows "Welcome, User"
- URL changes to "/dashboard"
- Session cookie is created

Postconditions:
- User is authenticated
- User remains logged in until logout

Test Data:
- Email: user@example.com
- Password: SecurePass123!
```

### Example 2: Data Validation

```
Test Case ID: TC-DATA-005
Title: System rejects email with invalid format
Priority: High

Preconditions:
- Signup form is open
- Fields empty

Steps:
1. Click on email input field
2. Enter "invalid.email.format"
3. Click elsewhere to trigger validation
4. Observe error message

Expected Results:
- Error message displays: "Please enter valid email format"
- Field highlighted in red
- Submit button remains disabled
- Form not submitted

Validation Rules Tested:
- ✓ Missing @ symbol
- ✓ Missing domain
- ✓ Invalid domain structure

Related Tests:
- TC-DATA-006: Valid email formats accepted
```

### Example 3: Integration

```
Test Case ID: TC-INT-010
Title: Payment processing with third-party gateway
Priority: High

Preconditions:
- User logged in
- Cart contains items ($99.99 total)
- Payment gateway (Stripe) is operational

Steps:
1. Navigate to checkout
2. Click "Proceed to Payment"
3. Select credit card payment
4. Enter card: 4242424242424242
5. Enter expiry: 12/25
6. Enter CVC: 123
7. Click "Pay Now"
8. Wait for processing

Expected Results:
- Payment gateway form loads
- Submission succeeds
- Order confirmed with ID OD-12345
- Confirmation email sent
- Inventory decremented
- Payment logged in database

Postconditions:
- Order status: "Paid"
- Customer can track in dashboard
```

## AI-Generated vs Manual Tests

### AI-Generated Tests

**Strengths:**
- ✅ Rapid generation from requirements
- ✅ Consistent format and naming
- ✅ Cover happy path + negative cases
- ✅ Easy to regenerate if requirements change

**Limitations:**
- ⚠️ May miss edge cases
- ⚠️ Limited to obvious test scenarios
- ⚠️ Can't test performance requirements
- ⚠️ May lack real-world context

### Manual Tests (After AI Generation)

**Recommended Additions:**
1. **Security edge cases** - SQL injection, XSS, CSRF
2. **Performance tests** - Load testing, stress testing
3. **Integration scenarios** - Multi-system workflows
4. **Real-world workflows** - Actual user behavior patterns
5. **Accessibility tests** - Screen reader, keyboard navigation
6. **Mobile-specific tests** - Touch interactions, responsive design
7. **Error recovery** - Network failures, timeouts

### Best Practice: Hybrid Approach

```
AI GENERATES:
- Basic happy path tests
- Error condition tests
- Input validation tests
- CRUD operations

YOU ADD:
- Security pen-tests
- Performance/load tests
- Complex integration scenarios
- Real-world edge cases
- Accessibility tests
- Mobile user flows
```

## Maintenance Guide

### Updating Tests When Requirements Change

```
BEFORE:
Req: "User can set profile visibility to public or private"
Test: Verify both options appear in settings

AFTER:
Req: "User can set profile visibility to public, private, or friends-only"
Update Test: Verify ALL THREE options appear in settings
Add Test: Verify friends-only permission rules work
```

### Test Review Checklist

Before adding test to suite, verify:

- [ ] **Unique ID** - Test case has unique, consistent ID
- [ ] **Clear Title** - Anyone can understand test purpose
- [ ] **Repeatable** - Same input always gives same result
- [ ] **Independent** - Test doesn't depend on other tests
- [ ] **Complete** - All steps clearly described
- [ ] **Correct Expected Results** - Aligned with requirements
- [ ] **Proper Classification** - Category and priority assigned
- [ ] **No Duplicates** - Similar test doesn't already exist
- [ ] **Maintainable** - Easy to update if requirements change

### Test Deprecation

When tests are no longer needed:

```
❌ DEPRECATED TESTS:
- Removed login with deleted user accounts → User account deletion tested separately
- Old API endpoint v1 tests → v2 replaces v1
- Legacy feature tests → Feature discontinued in v2.0

✅ REPLACEMENT STRATEGY:
- Map old tests to new equivalents
- Update references in documentation
- Archive old tests for historical records
- Document deprecation reason
```

## Advanced Topics

### Parameterized Testing

```
Test: Email validation with multiple inputs

Test Data:
| Input | Valid | Expected |
|-------|-------|----------|
| user@domain.com | Yes | Accept |
| invalid.email | No | Reject |
| test@example.co.uk | Yes | Accept |
|  | No | Reject (empty) |

Single test runs 4 times with different data
```

### Test Hooks & Fixtures

```
BEFORE EACH TEST:
- Create fresh test database
- Initialize test user
- Reset application state

AFTER EACH TEST:
- Clean up test data
- Close database connections
- Release resources

BEFORE ALL TESTS:
- Set up test environment
- Initialize external services

AFTER ALL TESTS:
- Clean up resources
- Generate coverage report
```

### Test Data Management

```
✅ GOOD PRACTICES:
- Use unique identifiers (email + timestamp)
- Clean up after each test
- Don't hardcode production data
- Use test data factories

❌ POOR PRACTICES:
- Hardcode common test data (shared across tests)
- Never clean up (leaves garbage data)
- Use production database for tests
- Interdependent test data
```
